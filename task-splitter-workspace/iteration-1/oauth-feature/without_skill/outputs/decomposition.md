# Decomposition Summary
- Goal: Add Google and GitHub OAuth2 login to an existing Express.js REST API that uses Passport.js session-based auth and PostgreSQL. The existing email/password login must continue working. The backend needs new Passport strategies, OAuth callback endpoints, and user-linking logic. The React frontend needs OAuth login buttons and callback handling. The users table needs new columns to store OAuth provider data. Testing (Jest) and CI/CD (GitHub Actions deploying to AWS ECS) must be updated accordingly.
- Likely Affected Areas:
  - Database: `users` table schema (new OAuth columns)
  - Backend: Passport.js configuration, OAuth strategy setup, auth routes, user model/queries
  - Frontend: Login and signup pages (OAuth buttons, callback route)
  - Infra/CI: Environment variables for OAuth client IDs/secrets, GitHub Actions workflow
- Main Workstreams:
  - Database schema migration (additive columns for OAuth)
  - Backend OAuth strategy and route implementation
  - Frontend OAuth button and callback integration
  - Testing across all layers
  - CI/CD and environment configuration
- Highest-Risk Changes:
  - Schema migration on the users table (must be backward-compatible, non-breaking)
  - Account linking logic (matching OAuth email to existing email/password accounts)
  - Session handling across OAuth and local strategies (must coexist)
  - CORS configuration for OAuth callback redirects between the API and app.example.com
- Assumptions:
  - Passport.js is already configured with `passport-local` strategy and express-session middleware
  - The PostgreSQL `users` table currently has columns: `id`, `email`, `password_hash`, `created_at`
  - The React frontend is a separate SPA served at `app.example.com`, communicating with the API via HTTP
  - OAuth provider credentials (Google Client ID/Secret, GitHub Client ID/Secret) will be provisioned outside this work and supplied as environment variables
  - The existing session-based auth mechanism (cookies/sessions) will also be used for OAuth-authenticated users (i.e., OAuth login creates a session, not a JWT)
  - `password_hash` is nullable or will need to become nullable to support OAuth-only users
- Open Questions:
  - Is `password_hash` currently nullable in the database? If not, the migration must alter the column to allow NULL for OAuth-only users.
  - What migration tool is used (Knex migrations, node-pg-migrate, raw SQL scripts, a custom runner)? The migration card assumes a migration framework exists but does not invent a specific one.
  - Should OAuth-only users be able to later set a password (and vice versa, should email/password users be able to link OAuth)? This decomposition assumes basic account linking by email but defers advanced linking flows to follow-up.
  - Are there rate-limiting or abuse-prevention concerns for the new OAuth endpoints?
  - What is the session store (in-memory, Redis, PostgreSQL)? OAuth login will use the same session store.
- Execution Waves:
  - Wave 1: TASK-1 (discovery and codebase audit)
  - Wave 2 (after TASK-1): TASK-2 (database migration), TASK-3 (backend OAuth config and dependencies)
  - Wave 3 (after TASK-2, TASK-3): TASK-4 (backend OAuth routes and account linking logic)
  - Wave 4 (after TASK-4): TASK-5 (frontend OAuth buttons and callback handling)
  - Wave 5 (after TASK-4, TASK-5): TASK-6 (testing across backend and frontend)
  - Wave 6 (after TASK-6): TASK-7 (CI/CD environment config and deployment)
  - Within each wave, independent tasks can run in parallel.

## TASK-4: Implement OAuth routes, callback handlers, and account-linking logic
- Goal: Add the Express routes for initiating and completing Google and GitHub OAuth flows, implement the `findOrCreateOAuthUser` database logic that links OAuth identities to user records, and update Passport serialization to handle OAuth users.
- Context Anchor: This is the core backend implementation task. It connects the strategies from TASK-3 to actual routes and the database schema from TASK-2, completing the backend OAuth flow end-to-end.
- In Scope:
  - Add routes (estimated: in the auth routes file identified in TASK-1):
    - `GET /auth/google` -- initiates Google OAuth flow via `passport.authenticate('google', { scope: ['profile', 'email'] })`
    - `GET /auth/google/callback` -- handles Google callback, creates session, redirects to frontend
    - `GET /auth/github` -- initiates GitHub OAuth flow via `passport.authenticate('github', { scope: ['user:email'] })`
    - `GET /auth/github/callback` -- handles GitHub callback, creates session, redirects to frontend
  - Implement `findOrCreateOAuthUser` function:
    - Look up user by `oauth_provider` + `oauth_provider_id`
    - If found, return existing user
    - If not found, look up user by email (to link OAuth to existing email/password account)
    - If email match found, update that user's `oauth_provider`, `oauth_provider_id`, `display_name`, `avatar_url` and return
    - If no match, create a new user with `email`, `oauth_provider`, `oauth_provider_id`, `display_name`, `avatar_url`, and `password_hash` = NULL
  - Update Passport `serializeUser` / `deserializeUser` if needed to handle users who have no `password_hash`
  - On successful OAuth callback, redirect to the React frontend (e.g., `app.example.com/auth/callback`) with session established
  - Handle OAuth errors gracefully (redirect to frontend with error query param)
  - Update CORS configuration if needed to allow the OAuth redirect flow between the API and `app.example.com`
- Non-Goals:
  - Frontend changes (TASK-5)
  - Allowing a user to link multiple OAuth providers simultaneously (follow-up)
  - Implementing OAuth token refresh (follow-up; not needed for login-only flow)
  - Revoking OAuth tokens on logout (follow-up)
- Dependencies: TASK-2 (OAuth columns must exist), TASK-3 (strategies must be configured)
- Parallelizable: No (depends on both TASK-2 and TASK-3)
- Can Land Independently: Yes (endpoints are functional; frontend buttons not yet present but routes can be tested directly)
- Change Type: backend
- Risk Level: medium
- Compatibility:
  - API: additive backward-compatible (new endpoints only; no changes to existing endpoints)
  - Data: additive schema (writes to new columns added in TASK-2)
  - Rollout: independent (new endpoints; existing auth unaffected)
- Acceptance Criteria:
  - `GET /auth/google` redirects the user to Google's OAuth consent screen
  - `GET /auth/google/callback` handles the Google callback, creates or links the user, establishes a session, and redirects to the frontend
  - `GET /auth/github` redirects the user to GitHub's OAuth authorization page
  - `GET /auth/github/callback` handles the GitHub callback, creates or links the user, establishes a session, and redirects to the frontend
  - A brand-new OAuth user (no matching email) gets a new row in `users` with `password_hash` = NULL
  - An OAuth user whose email matches an existing email/password user gets linked to that existing account (OAuth columns populated on the existing row)
  - Existing email/password login (`POST /auth/login` or equivalent) continues to work identically

  - `curl -v http://localhost:<port>/auth/github` (best-effort: should return a 302 redirect to github.com/login/oauth/authorize)
  - `npm test` (best-effort: confirm existing tests still pass)

  - Google OAuth requires the app to go through a verification process for production use; ensure this is initiated before production deployment
  - GitHub OAuth apps have separate callback URL configurations per environment; ensure these are set correctly
  - If the ECS task definition is managed by Terraform, CloudFormation, or CDK, the infra-as-code files must be updated rather than making manual console changes
- Scope Check: Fits one focused PR (CI/CD and infra config changes only)
