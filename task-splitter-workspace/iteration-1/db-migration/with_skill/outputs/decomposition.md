# Decomposition Summary

- **Goal:** Migrate the primary `users` table (12M rows) from RDS MySQL 8.0 to RDS PostgreSQL 15 with zero downtime. Three consuming services (Node.js API, Python/Celery worker, Rails admin dashboard) must be updated. The table has foreign key relationships with `orders`, `sessions`, and `audit_logs`. There is no existing PostgreSQL infrastructure on AWS; it must be provisioned from scratch.

- **Likely Affected Areas:**
  - AWS infrastructure (RDS PostgreSQL provisioning, networking, security groups, IAM)
  - Database schema and data (users, orders, sessions, audit_logs -- at minimum the users table and its FK relationships)
  - Node.js main API service (database driver, ORM/query layer, connection config)
  - Python/Celery background job processor (database driver, ORM/query layer, connection config)
  - Rails admin dashboard (database adapter, ActiveRecord config, queries)
  - CI/CD pipelines and deployment configurations
  - Monitoring, alerting, and observability stacks

- **Main Workstreams:**
  - Infrastructure provisioning (RDS PostgreSQL, networking, secrets)
  - Schema translation and validation (MySQL to PostgreSQL DDL)
  - Data replication pipeline (initial load + continuous replication)
  - Service-by-service dual-read/dual-write adaptation
  - Read cutover and write cutover per service
  - Data validation and reconciliation
  - Legacy MySQL cleanup

- **Highest-Risk Changes:**
  - Zero-downtime cutover of writes from MySQL to PostgreSQL across 3 services
  - Data integrity during continuous replication of 12M+ rows with live writes
  - Behavioral differences between MySQL 8.0 and PostgreSQL 15 (collation, type coercion, implicit casting, auto-increment vs. sequences, timestamp handling)
  - Foreign key relationships -- orders, sessions, audit_logs must either migrate together or use a compatibility strategy
  - Coordinated rollout across 3 services in different tech stacks

- **Assumptions:**
  - The `orders`, `sessions`, and `audit_logs` tables will also need to migrate to PostgreSQL (or at minimum their FK constraints referencing `users` must be handled). This decomposition assumes they migrate together or that FK enforcement is temporarily relaxed during the transition.
  - AWS DMS (Database Migration Service) is available and suitable for continuous replication from MySQL to PostgreSQL.
  - Each service has its own database connection configuration that can be changed independently (e.g., via environment variables or config files).
  - A feature flag or configuration toggle mechanism exists or can be introduced in each service to control read/write routing.
  - The team has the ability to provision new RDS instances and modify security groups in their AWS account.

- **Open Questions:**
  - Do the `orders`, `sessions`, and `audit_logs` tables also migrate to PostgreSQL, or do they remain in MySQL with cross-database references removed?
  - Are there MySQL-specific features in use (stored procedures, triggers, views, fulltext indexes, generated columns, JSON operators) that need PostgreSQL equivalents?
  - What ORM/query libraries does each service use? (e.g., Sequelize/Knex for Node.js, SQLAlchemy for Python, ActiveRecord for Rails)
  - Are there any MySQL-specific SQL constructs in application code (e.g., `INSERT ... ON DUPLICATE KEY UPDATE`, backtick quoting, `IFNULL`, `GROUP_CONCAT`, `LIMIT` with offsets)?
  - What is the current replication or backup strategy for the MySQL instance?
  - Are there compliance or data residency requirements affecting the new PostgreSQL instance?
  - What is the acceptable replication lag threshold for cutover?
  - Is there a maintenance window preference, even though zero downtime is required?

- **Execution Waves:**
  - Wave 1: TASK-01, TASK-02 -- discovery and infrastructure provisioning
  - Wave 2 (after TASK-01, TASK-02): TASK-03, TASK-04 -- schema translation and replication pipeline
  - Wave 3 (after TASK-03, TASK-04): TASK-05, TASK-06, TASK-07 -- service dual-read/dual-write adaptation (parallelizable across services)
  - Wave 4 (after TASK-04, TASK-05, TASK-06, TASK-07): TASK-08 -- data validation and reconciliation
  - Wave 5 (after TASK-08): TASK-09 -- read cutover (all services to PostgreSQL)
  - Wave 6 (after TASK-09): TASK-10 -- write cutover (all services to PostgreSQL)
  - Wave 7 (after TASK-10): TASK-11 -- decommission replication and legacy cleanup
  - Within each wave, independent tasks can run in parallel.

- **Goal:** Configure AWS DMS to perform a full initial load of all rows from `users`, `orders`, `sessions`, and `audit_logs` tables from MySQL to PostgreSQL, followed by continuous change data capture (CDC) to keep PostgreSQL in near-real-time sync with MySQL.
- **Context Anchor:** This is the data replication task in Wave 2. Continuous replication is the foundation for zero-downtime migration -- it keeps PostgreSQL current while services are gradually cut over. Depends on TASK-02 (target instance) and TASK-03 (target schema).
- **In Scope:**
  - AWS DMS replication instance provisioning (appropriate instance class for 12M rows + ongoing CDC)
  - Source endpoint configuration (RDS MySQL 8.0)
  - Target endpoint configuration (RDS PostgreSQL 15)
  - DMS task configuration: full-load + CDC mode
  - Table mappings for `users`, `orders`, `sessions`, `audit_logs`
  - Transformation rules for any type or value differences identified in TASK-01/TASK-03
  - Enabling MySQL binary logging if not already enabled (required for CDC)
  - Monitoring DMS task health and replication lag via CloudWatch
  - Validation that initial load completes successfully with row count parity
- **Non-Goals:**
  - Modifying application code
  - Cutting over any service reads or writes
  - Performance tuning of the PostgreSQL instance
- **Dependencies:** TASK-02, TASK-03
- **Parallelizable:** No (must run after TASK-03 completes schema deployment)
- **Can Land Independently:** Yes
- **Change Type:** infra
- **Risk Level:** high
- **Compatibility:**
  - API: none
  - Data: backfill
  - Rollout: independent
- **Acceptance Criteria:**
  - DMS replication instance is running
  - Full load completes: row counts match between MySQL and PostgreSQL for all 4 tables
  - CDC is active: inserts, updates, and deletes on MySQL are reflected in PostgreSQL within acceptable lag (target: < 5 seconds under normal load)
  - DMS CloudWatch metrics are visible (replication lag, CDC latency, errors)
  - A data spot-check of 100+ random rows confirms field-level parity between MySQL and PostgreSQL
  - Sequences in PostgreSQL are set to values beyond the current max IDs to prevent conflicts when writes switch over
- **Verification Prerequisites:**
  - AWS DMS service access
  - MySQL binary logging enabled (`binlog_format = ROW`)
  - Network connectivity between DMS replication instance, MySQL, and PostgreSQL
- **Verification Commands:**

  - `aws dms describe-table-statistics --replication-task-arn <arn>` (best-effort)
  - Row count comparison: `SELECT COUNT(*) FROM users;` on both MySQL and PostgreSQL (best-effort)
  - `psql -c "SELECT last_value FROM users_id_seq;"` vs. `mysql -e "SELECT MAX(id) FROM users;"` -- verify sequence alignment (best-effort)
- **Constraints:**

  - DMS replication instance must be in the same VPC or a peered VPC
  - Replication must be continuously monitored for errors; any LOB (large object) columns may require special handling
  - Must not degrade MySQL production performance -- DMS read load should be monitored
- **Rollout / Rollback Notes:**
  - Rollback: stop and delete the DMS task and replication instance. PostgreSQL data can be truncated. No production impact since no services read from PostgreSQL yet.
  - If MySQL performance degrades, throttle DMS task or resize replication instance.
- **Notes/Risks:**
  - DMS has known limitations with certain MySQL types (e.g., JSON columns, spatial types, very large BLOBs). These should be identified in TASK-01.
  - If `binlog_format` is not `ROW`, changing it on the RDS parameter group requires a reboot of the MySQL instance. This must be coordinated.
  - Initial full load of 12M rows may take significant time depending on row size and instance class. Estimate and plan accordingly.
  - LOB columns may require `FullLobMode` or `LimitedLobMode` configuration.
- **Scope Check:** Fits one focused PR (IaC for DMS resources + operational verification).

## [TASK-10] Cut over all service writes to PostgreSQL and stop DMS replication
