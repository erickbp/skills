# Decomposition Summary

- **Goal:** Migrate a single-tenant SaaS analytics platform to a shared-infrastructure multi-tenant architecture where all tenants share the same PostgreSQL database and application instances, with strict data isolation. The system comprises a Next.js frontend, FastAPI backend, PostgreSQL database (approximately 40 tables), Redis caching layer, and Celery async job workers. The platform has approximately 200 API endpoints, 15 background job types, and must scale from 23 tenants to 500+ within a year.

- **Likely Affected Areas:**
  - PostgreSQL schema (all 40 tables need tenant scoping)
  - FastAPI backend (all approximately 200 endpoints need tenant context injection and filtering)
  - Celery workers (all 15 job types need tenant-aware execution)
  - Redis caching (cache keys and isolation per tenant)
  - Next.js frontend (tenant resolution, routing, session context)
  - Alembic migrations (tenant column additions, index creation, data backfill)
  - Authentication and session management
  - Test suites (pytest, Cypress)
  - Infrastructure and deployment configuration

- **Main Workstreams:**
  1. Tenant data model and identity foundation
  2. Database schema migration (add tenant_id columns, indexes, constraints)
  3. Backend tenant context middleware and data-access-layer enforcement
  4. API endpoint tenant scoping
  5. Celery job tenant isolation
  6. Redis cache tenant isolation
  7. Frontend tenant resolution and context
  8. Data migration / backfill for existing 23 tenants
  9. Read/write path cutover
  10. Test infrastructure updates
  11. Observability and operational tooling

- **Highest-Risk Changes:**
  - Schema migration on 40 tables with live data (locks, downtime risk)
  - Data backfill correctness for existing 23 tenants
  - Incomplete tenant filtering leading to cross-tenant data leakage (security/privacy critical)
  - Cache poisoning or cross-tenant cache reads in Redis
  - Celery jobs processing data across tenant boundaries
  - Performance regression at 500+ tenants due to missing or suboptimal indexes
  - Migration rollback complexity once write-path switches to multi-tenant

- **Assumptions:**
  - Each existing single-tenant deployment has its own database with the same schema, so data can be mapped 1:1 to a tenant_id
  - A "tenants" table or equivalent does not yet exist in the shared database
  - Row-Level Security (RLS) in PostgreSQL is a viable enforcement layer (to be confirmed in discovery)
  - The existing authentication system can be extended to carry tenant identity (e.g., JWT claims or session metadata)
  - Alembic is the sole migration tool and supports the required migration patterns
  - The subdomain-based routing model will be preserved (tenant resolution by subdomain)
  - There is no existing multi-tenant middleware or tenant context in the codebase

- **Open Questions:**
  - Is there an existing user-to-tenant mapping, or are users currently implicitly scoped by deployment?
  - Are there any tables that are intentionally shared across tenants (e.g., lookup/reference data)?
  - What is the largest table by row count, and can it tolerate an ALTER TABLE ADD COLUMN in a single migration without downtime?
  - Does the FastAPI application use an ORM (SQLAlchemy) or raw SQL? This affects the tenant-filtering strategy.
  - Are Celery jobs currently stateless with respect to tenant, or do they read tenant-specific configuration?
  - Is there a CI/CD pipeline that runs Alembic migrations automatically on deploy?
  - What is the acceptable downtime window, if any, for the migration cutover?
  - Are there any compliance requirements (SOC2, GDPR, HIPAA) that mandate specific data isolation guarantees beyond application-level filtering?

- **Execution Waves:**
  - **Wave 1:** TASK-01, TASK-02 — Discovery and tenant data model foundation
  - **Wave 2 (after TASK-01, TASK-02):** TASK-03, TASK-04, TASK-05 — Backend middleware, schema migration planning, and Redis isolation design
  - **Wave 3 (after TASK-03, TASK-04):** TASK-06, TASK-07, TASK-08 — Schema migration execution, DAL tenant enforcement, Celery tenant isolation
  - **Wave 4 (after TASK-06, TASK-07):** TASK-09, TASK-10, TASK-11 — API endpoint scoping, frontend tenant context, data backfill
  - **Wave 5 (after TASK-09, TASK-10, TASK-11):** TASK-12, TASK-13 — Read/write cutover, test infrastructure overhaul
  - **Wave 6 (after TASK-12, TASK-13):** TASK-14, TASK-15 — Observability/operational tooling, end-to-end validation and staged rollout
  - Within each wave, independent tasks can run in parallel.

- **In Scope:**

  - Create the SQLAlchemy model (or equivalent) for the Tenant entity (estimated location: models directory)
  - Seed the tenants table with the existing 23 tenant records (slug derived from current subdomains)
  - Define the tenant_id type (UUID vs integer) and document the decision
  - Create a utility function for resolving tenant from subdomain (to be used by middleware in TASK-03)
  - Create a tenant lookup cache strategy (in-memory with TTL, or Redis-backed)
- **Non-Goals:**
  - Adding tenant_id to any other tables (that is TASK-06)
  - Building the request middleware (that is TASK-03)
  - Modifying any API endpoints
  - Building Row-Level Security policies
- **Dependencies:** None (can start in parallel with TASK-01 if tenant table design is straightforward; ideally informed by TASK-01 findings)
- **Parallelizable:** Yes (with TASK-01)
- **Can Land Independently:** Yes
- **Change Type:** schema / backend
- **Risk Level:** low
- **Compatibility:**
  - API: none (no endpoint changes)
  - Data: additive schema (new table only)
  - Rollout: independent
- **Acceptance Criteria:**
  - The `tenants` table exists in the database after running the Alembic migration
  - The migration is reversible (downgrade drops the table)
  - A Tenant model is importable from the application's model layer
  - A `resolve_tenant_by_subdomain(subdomain: str) -> Tenant` utility function exists and is tested
  - All 23 existing tenants are seeded (via migration data seed or a separate seed script)
  - The tenant_id type decision is documented in a comment or docstring
- **Verification Prerequisites:**
  - PostgreSQL database accessible
  - Alembic configured and runnable
- **Verification Commands:**

  - `alembic downgrade -1 && alembic upgrade head` (best-effort: verify reversibility)
  - `pytest tests/ -k "tenant" -v` (best-effort: run tenant-related tests)
- **Constraints:**

  - The tenants table must support soft-delete or status-based deactivation (not hard delete)
  - Slug/subdomain must be unique and validated

  - This migration is purely additive and safe to deploy independently.
- **Notes/Risks:**
  - The tenant_id type decision (UUID vs integer) has system-wide implications for index size and join performance at 500+ tenants. UUID is recommended for external exposure; integer for internal performance. Consider using both (integer PK + UUID external ID).
- **Scope Check:** Fits one focused PR

## [TASK-03] Implement tenant context middleware and request-scoped tenant injection

- **Goal:** Build the FastAPI middleware that resolves the tenant from each incoming request (via subdomain), validates it, and makes the tenant context available to all downstream request handlers via a request-scoped dependency or context variable.
- **Context Anchor:** This is the first task in Wave 2. It provides the mechanism by which every API endpoint will know which tenant is being served. Without this, endpoint-level tenant scoping cannot proceed.
- **In Scope:**
  - FastAPI middleware or dependency that extracts subdomain from the request Host header
  - Tenant resolution using the utility from TASK-02
  - Request-scoped tenant context (e.g., using Python contextvars, Starlette request state, or FastAPI dependency injection)
  - Error handling for unknown/invalid/deactivated tenants (return 404 or 403)
  - Unit tests for the middleware
  - Integration test verifying tenant resolution end-to-end on a test endpoint
- **Non-Goals:**
  - Modifying existing API endpoints to use the tenant context (that is TASK-09)
  - Database query filtering (that is TASK-07)
  - Celery job tenant context (that is TASK-08)
  - Authentication changes beyond carrying tenant_id
- **Dependencies:** TASK-02 (tenant model and resolution utility)
- **Parallelizable:** Yes (with TASK-04, TASK-05)
- **Can Land Independently:** Yes (middleware can be deployed without existing endpoints consuming it)
- **Change Type:** backend
- **Risk Level:** medium
- **Compatibility:**
  - API: additive backward-compatible (middleware is additive; existing endpoints unaffected until they opt in)
  - Data: none
  - Rollout: independent (can deploy without any endpoint using it yet)
- **Acceptance Criteria:**
  - Every incoming HTTP request has a resolved tenant available in the request context
  - Requests to unknown subdomains return a 404 (or configurable error)
  - Requests to deactivated tenants return a 403
  - The tenant context is accessible from any FastAPI route handler or dependency
  - The middleware does not break any existing endpoint behavior
  - Unit tests cover: valid tenant, unknown tenant, deactivated tenant, missing Host header
  - Performance: tenant resolution adds less than 5ms latency (cached lookup)
- **Verification Prerequisites:**
  - FastAPI application runnable locally
  - Tenants table seeded (from TASK-02)
- **Verification Commands:**
  - `pytest tests/ -k "middleware" -v` (best-effort)
  - `pytest tests/ -k "tenant_context" -v` (best-effort)
  - `curl -H "Host: acme.analytics.io" http://localhost:8000/health` (best-effort: verify tenant resolution)
  - `curl -H "Host: unknown.analytics.io" http://localhost:8000/health` (best-effort: verify 404)
- **Constraints:**
  - Must not break any existing endpoint behavior (backward compatible)
  - Must handle concurrent requests with different tenants safely (no thread/async context leakage)
  - Tenant resolution must be cached to avoid a database query per request
- **Rollout / Rollback Notes:**
  - Rollback: remove the middleware registration. No data changes involved.
  - Can be feature-flagged: middleware resolves tenant but does not enforce it until a flag is enabled.
- **Notes/Risks:**
  - Context leakage between async requests is a critical security risk. If using contextvars, ensure proper scoping per request lifecycle.
  - If the application is deployed behind a reverse proxy that strips the Host header, the subdomain resolution strategy may need adjustment.
- **Scope Check:** Fits one focused PR

- **Non-Goals:**

- **Dependencies:** TASK-02 (tenant_id type definition)

- **Change Type:** backend

## [TASK-07] Implement tenant-scoped data access layer (DAL) enforcement

- **In Scope:**

  - If raw SQL: create a query builder wrapper or template that injects tenant_id filtering
  - Create a `TenantScopedModel` mixin or base class that all tenant-scoped models inherit from
  - Integrate with the request-scoped tenant context from TASK-03
  - Implement a bypass mechanism for admin/system operations that legitimately need cross-tenant access (explicitly opt-in, logged)
  - Unit tests verifying: queries are filtered, cross-tenant access is denied by default, bypass works when explicitly requested
  - Security test: verify that a query without tenant context raises an error (fail-closed)
- **Non-Goals:**
  - Updating individual model definitions to inherit from TenantScopedModel (that is part of TASK-09)
  - PostgreSQL Row-Level Security policies (that is TASK-12)
  - Modifying API endpoints
- **Dependencies:** TASK-03 (tenant context middleware), TASK-06 (tenant_id columns exist)
- **Parallelizable:** Yes (with TASK-08)
- **Can Land Independently:** Behind flag (DAL enforcement can be toggled on/off during transition)
- **Change Type:** backend
- **Risk Level:** high
- **Compatibility:**
  - API: none (internal change)
  - Data: none (query-level change only)
  - Rollout: feature-flagged
- **Acceptance Criteria:**
  - All queries through the DAL on tenant-scoped models include a tenant_id filter
  - Queries without an active tenant context raise an explicit error (not silently return all data)
  - An explicit bypass mechanism exists for admin operations, and its use is logged
  - Unit tests verify tenant filtering, fail-closed behavior, and bypass mechanism
  - No existing tests break when the feature flag is off
- **Verification Prerequisites:**
  - Database with tenant_id columns (from TASK-06)
  - Tenant context middleware deployed (from TASK-03)
  - Test database with multi-tenant test data
- **Verification Commands:**

  - `pytest tests/ -k "data_access" -v` (best-effort)
  - `pytest tests/ -v` (best-effort: full suite regression)
- **Constraints:**
  - Fail-closed: missing tenant context must error, never return unscoped data
  - Must not introduce N+1 query patterns or measurable performance regression
  - Bypass mechanism must require explicit opt-in and produce audit log entries

- **Notes/Risks:**
  - This is the most security-critical component. Incomplete filtering leads directly to cross-tenant data leakage.
  - If the application uses raw SQL in some places, those queries will bypass the DAL filter. TASK-01 discovery should have identified these; they need individual attention.
  - Performance: adding a WHERE clause to every query should be negligible with proper indexes (from TASK-06), but monitor query plans for the most critical queries.

- **Goal:** Ensure all 15 Celery background job types execute within a tenant context, with tenant_id passed as a job parameter and tenant-scoped database access enforced during job execution.

- **In Scope:**

  - Set up tenant context (contextvars or equivalent) at the start of each job execution

  - Update all 15 job types to accept and pass tenant_id (or document exact list for TASK-09 to handle if tightly coupled with endpoint changes)
  - Ensure job retries preserve tenant context
  - Ensure job logging includes tenant_id for observability
  - Unit tests for the base task class
  - Integration test for at least one representative job type
- **Non-Goals:**
  - Tenant-based job queue separation (e.g., separate queues per tenant) — this is a future optimization
